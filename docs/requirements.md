# 就活管理アプリ shukatsu-app 要件定義書

最終更新: 2026-05-01

## 1. 目的

就職活動における「応募企業」「選考ステータス」「各種締切」を 1 か所で管理し、
抜け漏れなく就活を進められるようにする個人向け Web アプリ。

副次目的として、Next.js + TypeScript + React の習得を兼ねる。

## 2. 利用者

- 主利用者: 開発者本人 (1 名想定)
- 同時に複数人で使う想定なし。マルチアカウント機能は MVP に含めない。

## 3. 機能一覧 (MVP)

優先度は P1 (必須) > P2 (重要) > P3 (あれば嬉しい)。

| # | 機能 | 概要 | 優先度 |
|---|------|------|--------|
| F1 | 企業登録 | 企業名・URL・業界・自由メモ等を新規登録 | P1 |
| F2 | 企業一覧表示 | 登録済み企業を一覧表示。各行に選考ステータスと直近の締切を表示 | P1 |
| F3 | 選考ステータス管理 | 各企業の選考状態を変更できる (下記「選考ステータス一覧」参照)。本選考とインターン選考の両方に対応 | P1 |
| F4 | 締切日登録 | 企業ごとに「ES締切」「面接日」など複数のタスクと日時を登録 | P1 |
| F5 | 絞り込み・並び替え | ステータス・締切日で一覧を絞り込み/並び替え | P2 |
| F6 | メモ機能 | 企業ごとに自由記入のメモ (面接で聞かれたこと等) | P2 |
| F7 | カレンダー連携 | 締切・面接日を Google カレンダーに出力 (まずは ics ダウンロードでも可) | P3 |
| F8 | 通知機能 | 締切が近づいたらブラウザ通知で知らせる (Web Notifications API) | P3 |

### 3.1 選考ステータス一覧

選考種別 (本選考 / インターン) を 1 つ持ち、その下に状態を持つ。

**インターン選考**
- 興味あり (未応募)
- ES提出
- Webテスト
- コーディングテスト
- グループディスカッション
- 一次面接
- 二次面接
- 最終面接
- 参加決定
- 不採用
- 辞退

**本選考**
- 興味あり (未エントリー)
- エントリー済み
- ES提出
- Webテスト
- コーディングテスト
- グループディスカッション
- 一次面接
- 二次面接
- 最終面接
- 内定
- 不採用 (お祈り)
- 辞退

ステータスは「現在の状態」を 1 つ持つ。過去の経過は F4 の締切/タスクと F6 のメモで補完する。

### 3.2 企業とエントリーの関係

1 つの企業に対して複数のエントリー (例: 夏インターン応募 + 冬の本選考) を持てる。
データモデルは「企業 (Company)」と「エントリー (Application)」を分け、1 対 N の関係とする。詳細は `docs/design.md` のデータモデル節で定義する。

## 4. スコープ外 (MVP に含めないもの)

明示的に外す。後で迷わないため。

- マルチユーザー / 認証 (将来検討)
- スマホネイティブアプリ (Web のみ。レスポンシブ対応はする)
- 求人スクレイピング・自動取り込み
- AI による志望動機生成・添削
- チームでの共有機能

## 5. 非機能要件

### 5.1 性能

- 一覧画面の初回表示: 体感 1 秒以内 (ローカル想定)
- 登録企業数は最大でも数百件想定。DB 設計はそれを超えない範囲で最適化しない。

### 5.2 対応ブラウザ・端末

- 最新版 Chrome / Safari / Firefox
- PC とスマートフォン (iOS Safari) で表示崩れがないこと

### 5.3 セキュリティ

- MVP は本人のみ利用するためログイン機能なし。
- ただし将来クラウドにデプロイすることを想定し、API キー等は環境変数で管理する。
- 個人情報 (本人/企業担当者) を扱うため、公開リポジトリでもデータは含めない (`.gitignore`)。

### 5.4 データの永続化

- MVP はブラウザの `localStorage` (または `IndexedDB`) に保存する。サーバー DB は持たない。
- 将来クラウド DB (Supabase など) に移行する可能性を考え、データアクセス層を 1 か所にまとめておく (UI からは保存先の実装を意識しない設計)。
- ブラウザを変えるとデータが移らない点はトレードオフとして許容する。バックアップ用に JSON エクスポート/インポート機能を将来追加する余地を残す。

### 5.5 アクセシビリティ

- キーボード操作で主要機能 (登録・編集・削除) ができること
- 配色はコントラスト比 4.5:1 を満たす (WCAG AA 相当)

## 6. 開発上の方針 (学習目的の制約)

- 1 機能ずつ実装し、ブラウザで動作確認してから次へ進む
- 新概念が出たら、コード前に日本語で解説する
- 設計判断は `docs/design.md` に記録する

## 7. リリース計画 (ざっくり)

| フェーズ | 含む機能 | 完了条件 |
|----------|----------|----------|
| Phase 1 | F1, F2 | 企業を登録して一覧で見られる |
| Phase 2 | F3, F4 | 選考状態と締切を管理できる |
| Phase 3 | F5, F6 | 一覧の絞り込みとメモが使える |
| Phase 4 | F7, F8 | カレンダー連携 / 通知 |

Phase 1〜3 までを MVP とみなす。Phase 4 はその後の拡張扱い。
